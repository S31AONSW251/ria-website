# Prompt: Fix website visuals, replace images, and productionize

Purpose
- Audit the current site, replace legacy images with the new images provided in the workspace (user-provided folders).
- Ensure logos and brand assets are placed correctly and scaled.
- Improve visual design (layout, spacing, typography, colors) to look professional and production-ready.
- Optimize images (responsive sizes, srcset, webp where appropriate, lazy-loading) and update alt text.
- Preserve or archive AION content according to instructions; prepare the site for RIA launch date.

Primary goals
- Replace all old/low-quality images with the new images provided in the workspace (user-provided folders).
- Ensure logos and brand assets are placed correctly and scaled.
- Improve visual design (layout, spacing, typography, colors) to look professional and production-ready.
- Optimize images (responsive sizes, srcset, webp where appropriate, lazy-loading) and update alt text.
- Preserve or archive AION content according to instructions; prepare the site for RIA launch date.

Required inputs (fill before running)
- `images_folders`: list of source folders (example: ["New Image","image uplode","public/images/aion-uploaded"]).
- `logo_name_patterns`: array of filename patterns to treat as logos (example: ["brain","logo","ria"]).
- `launch_date`: optional launch/critical date (example: "2026-06-21").
- `preserve_aion_pages`: boolean — whether to keep AION pages as-is or archive/rename them.
- `branch_name`: git branch to create for changes (default: `fix/design-and-assets`).
- `create_pr`: boolean — whether to open a pull request when done.

Acceptance criteria
- All visible images on pages replaced by chosen new images or optimized variants.
- Layout improvements applied consistently across breakpoints (mobile/tablet/desktop).
- Lighthouse/performance score improved for images and basic metrics (or a clear summary of tradeoffs).
- A new git branch with commits and a PR (if requested) containing: asset replacements, style adjustments, image optimization, and a short changelog with screenshots.

Step-by-step instructions for the assistant (executable checklist)
1. Create and switch to `{{branch_name}}`.
2. Run a quick repo audit: list pages and templates that reference images (search for `<img`, `background-image`, `srcset`, CSS assets). Produce an images inventory (old -> new candidates).
3. Scan `images_folders` for candidate images. For each page slot (hero, card, logos, icons), automatically choose the best-fit new image by resolution and aspect ratio. If no suitable image exists, mark for user review.
4. Replace images with these rules:
   - Use responsive `srcset` with 1x/2x sizes for raster images.
   - Convert large images to `webp` (keep original file as backup in `/artifacts/asset-backups/` or similar).
   - Add `loading="lazy"` to non-critical images and ensure hero images are prioritized with `fetchpriority="high"` where applicable.
   - Add or update meaningful `alt` text (generate from file name if missing, but flag for review).
5. Update logo placement and header/footer branding:
   - Replace header logo with highest-quality logo file matching `logo_name_patterns`.
   - Ensure favicon/meta images updated if replacement assets exist.
6. Improve layout and visual polish:
   - Audit spacing, font sizes, and color contrast on key pages (home, hero, cards, CTA sections).
   - Apply small CSS refinements (consistent container widths, readable typography, larger CTAs, visual rhythm).
   - Keep changes minimal but coherent — do not refactor unrelated JS or backend code.
7. Accessibility & SEO quick checks:
   - Ensure images have `alt` attributes.
   - Check title/meta description present on key pages; add/update if obviously missing.
8. Performance & optimization:
   - Compress images to sensible targets (hero <= 300–500 KB dependent on resolution; thumbnails < 100 KB), preferring `webp` for modern browsers.
   - Generate lower-resolution/modern formats and add `srcset`.
9. Build & test:
   - Run the project's build (e.g., `npm run build` or `vite build`).
   - Run a lightweight Lighthouse or `lighthouse-ci` check and capture results.
   - Take before/after screenshots of key pages.
10. Commit changes with atomic messages (example: `fix(assets): replace legacy images with new uploads; polish hero & card styles`).
11. If `create_pr` is true, push the branch and open a PR with:
    - Summary of changes
    - Before/after screenshots
    - Image inventory (which old image was replaced by which new file)
    - Any outstanding items that need user input

Edge cases and safety
- Do not delete original images; copy them to an `artifacts/asset-backups/YYYYMMDD/` folder before replacing.
- If a selected new image is too small for a hero slot, either: (A) upscale with a note, (B) pick alternate, or (C) flag for user approval.
- If multiple images match a slot equally, prefer the one whose filename contains a brand keyword (e.g., `ria`, `brain`, `logo`).

Questions to ask the user before running (these should be asked if any required input is missing)
1. Confirm the source image folder(s) to use (exact path(s)).
2. Should AION pages be preserved/archived or updated to RIA? (options: preserve, archive, rename/brand as RIA)
3. Do you want automatic image format conversion to `webp` and lossy compression, or keep originals and only add `srcset`?
4. Confirm the launch date/time for RIA (e.g., `2026-06-21`), and whether I should add a launch banner or countdown.
5. Do you prefer a PR for review or direct commits to the main branch? (recommended: PR)

Ambiguous/weak parts in the raw brief (need clarification)
- "All new image change all old image I uploade" — confirm exact folders and whether some legacy images must be kept.
- "brain logo image" — which file(s) are canonical logos? Provide names or a preferred file.
- "Make my website big production type website" — define priorities: visual polish only, full redesign, or brand-alignment only.
